Activepieces 的工作是以一週為單位的 Sprints (衝刺) 為基礎，由於優先事項變化快速，Sprint 必須保持簡短以利適應。

## Sprints (衝刺)

Sprints 會在我們的 GitHub 帳號上公開分享。這讓每個人都能透明地看到我們正在進行的工作。

* 應該提前為 Sprint 建立一個 GitHub issue，概述各項變更。
* 每個**個人**都應該準備好關於下一個 Sprint 要負責的工作內容的具體建議。**如果你擔任工程角色，沒有人會指示你該開發什麼——這取決於你自己的驅動。**
* 團隊通常每週開會一次，共同挑選**優先事項**。
* 團隊中的每個人都應該參加 Sprint 規劃會議。
* 任何人都可以對 Sprint issue 在開始前或結束後發表評論。

## Pull Requests (拉取請求)

關於代碼審查 (code review)，我們有一些確保效率的準則：

- 儘早建立處於草稿狀態 (draft state) 的 pull request。
- 保持主動並審查他人的 pull requests。不要等別人要求你審查；這是你的責任。
- 為你的 pull request 僅指派一名審查者。
- 將 PR 加入目前的專案 (sprint)，以便我們在每個 sprint 結束時追蹤未合併的 PR。
- **PR 負責人**有責任在 PR 描述中起草測試場景。在審查時，審查者可以假設這些場景已經過測試，並提供額外的建議場景。
- 大型、未完成的功能應拆分為較小的任務，並持續合併到主分支 (main branch) 中。

## 規劃是每個人的工作。

每位工程師都有責任發現 bug/機會，並在 sprint 中提出，將其轉換為可執行的任務。
